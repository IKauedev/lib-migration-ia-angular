## Commands

### `config` — Configure AI provider

Opens an interactive wizard to select and configure the AI provider.

```bash
ng-migrate config               # Interactive wizard
ng-migrate config --show        # Display current configuration
ng-migrate config --reset       # Remove all saved settings
ng-migrate config --task-model  # Configure per-task models
```

**The wizard asks:**
- Which provider to use (Anthropic, OpenAI, Azure OpenAI, Google Gemini, OpenRouter, Ollama, etc.)
- API Key (displayed as password, not shown in terminal)
- Provider-specific fields (Azure endpoint, deployment, API version, etc.)
- Which model to use (e.g. `claude-opus-4-5`, `gpt-4o`, `gemini-2.0-flash`)

Configuration is saved to `~/.ng-migrate/config.json` and reused automatically.

**`--task-model` — per-task models:**

Configure different models for each operation type:

| Task | Description |
|---|---|
| `migration` | File migration (`migrate`) |
| `analysis` | File/project analysis (`analyze`) |
| `scan` | AI-powered scan (`scan --ai`) |
| `chat` | Interactive REPL (`repl`) |

### `scan` — Analyze project

Statically scans an AngularJS project and generates JSON analysis files that guide the migration.

```bash
ng-migrate scan                  # Analyze current folder
ng-migrate scan .                # Same
ng-migrate scan ./my-project     # Analyze specific folder
ng-migrate scan --ai             # Deep per-file AI analysis
ng-migrate scan --force          # Re-run analysis even if it already exists
ng-migrate scan --include-all    # Include files with no AngularJS patterns
ng-migrate scan --json           # Print full JSON to stdout
ng-migrate scan --no-save        # Don't save analysis files
```

**What is detected:**
- File type (controller, service, factory, directive, filter, component, template, routing)
- Complexity per file (low / medium / high) based on LOC and pattern weight
- AngularJS patterns present (`$scope`, `$http`, `ng-repeat`, `.directive()`, etc.)
- Injected dependencies (DI array, `$inject`, TypeScript constructors)
- Migration problems per file (`$compile`, `$rootScope`, `angular.element`, etc.)
- Suggested migration phase (1=services → 6=routing)
- Estimated hours per file and project total
- AngularJS version in use (`package.json`)
- All declared Angular modules
- All routes (`$routeProvider.when()` and `$stateProvider.state()`)
- Dependency graph between files

**After scanning, a library report is also displayed** from `package.json` — showing how many dependencies will be replaced, removed, or need manual work (see [libs command](#libs--library-migration-report)).

**3 files generated in the project:**

| File | Contents |
|---|---|
| `.ng-migrate-analysis.json` | Overall summary, file list, phases, migration plan |
| `.ng-migrate-registry.json` | Symbol map: `myUserSvc` → `MyUserService`; routes; modules |
| `.ng-migrate-deps-graph.json` | For each file: injected tokens and which files provide them |

### `libs` — Library migration report

Analyzes the `package.json` and generates a complete library migration report, categorizing each dependency.

```bash
ng-migrate libs                # Analyze current folder
ng-migrate libs ./my-project   # Analyze specific folder
ng-migrate libs --json         # JSON output (useful for CI/scripts)
ng-migrate libs --no-save      # Display only, don't save file
```

**5 classification categories:**

| Symbol | Category | Description |
|--------|----------|-------------|
| ✔ | **To replace** | Has an Angular 21 equivalent |
| ✖ | **To remove** | Obsolete — not needed in Angular |
| ⚠ | **Manual migration** | Requires human attention and adaptation |
| ? | **No mapping** | AngularJS-related but no known mapping |
| → | **Kept** | Not AngularJS — stays in the project |

**Examples of automatically mapped replacements:**

| AngularJS | Angular 21 |
|-----------|------------|
| `angular-ui-router` | `@angular/router` |
| `angular-animate` | `@angular/animations` |
| `angular-material` | `@angular/material` |
| `angular-translate` | `@ngx-translate/core` |
| `restangular` | `@angular/common/http` |
| `ui-select` | `@ng-select/ng-select` |

**Saved file:** `.ng-migrate-libs.json` in the project root.

> The `scan` command also displays an automatic library summary after scanning the project. Use `ng-migrate libs` for the full detailed report.

### `start` — Automated pipeline

Runs the full migration pipeline in 4 phases in a fully automated way: scan, plan, confirm, and migrate.

```bash
ng-migrate start                     # Pipeline in current folder
ng-migrate start ./my-project        # Pipeline in specific folder
ng-migrate start --only "src/**"     # Filter by glob
ng-migrate start --phase 1           # Run only phase 1 (services)
```

**The 4 pipeline phases:**

| Phase | What happens |
|---|---|
| **1/4 — Scan** | Detects all AngularJS files, complexity and dependencies |
| **2/4 — Plan** | Shows full summary: files, complexity, estimated hours, phase plan |
| **3/4 — Confirm** | Asks for confirmation before any changes |
| **4/4 — Migrate** | Runs `migrate-project` with the defined options |

> Unlike `migrate-project`, `start` is interactive — it displays the full plan and asks for confirmation before migrating.

### `migrate-project` — Migrate local project

Migrates all AngularJS files in a local folder to Angular 21, using the `scan` analysis to determine order and context.

```bash
ng-migrate migrate-project                    # Migrate current folder → <project>-angular21/
ng-migrate migrate-project .                  # Same
ng-migrate migrate-project ./src/app          # Migrate specific subfolder
ng-migrate migrate-project --in-place         # Migrate within the project itself
ng-migrate migrate-project -o ./output        # Set output folder
ng-migrate migrate-project --dry-run          # List what would be migrated without executing
ng-migrate migrate-project --concurrency 5    # 5 files in parallel
ng-migrate migrate-project --phase 1          # Migrate only services/factories
ng-migrate migrate-project --only "src/**"    # Filter by glob
ng-migrate migrate-project --skip-deps        # Don't update package.json
ng-migrate migrate-project --skip-install     # Don't run npm install at the end
ng-migrate migrate-project --fresh            # Ignore checkpoint and start fresh
ng-migrate migrate-project --clone https://github.com/user/repo  # Clone and migrate
```

> `migrate-project` **always scans the project before migrating**, displaying a full summary (files found, complexity, estimated hours, phase plan) before starting any changes.

**Options:**

| Option | Default | Description |
|---|---|---|
| `[folder]` | `.` (current) | AngularJS project root folder |
| `--clone <url>` | — | Clone git repository before migrating |
| `-o, --output <folder>` | `../<project>-angular21` | Output folder |
| `--in-place` | — | Migrate within the project itself |
| `-c, --concurrency <n>` | `3` | Number of files migrated in parallel |
| `--dry-run` | — | Show plan without executing |
| `--only <glob>` | — | Filter files by glob pattern |
| `--phase <n>` | — | Migrate only phase N (1–6) |
| `--skip-deps` | — | Don't update `package.json` |
| `--skip-install` | — | Don't run `npm install` at the end |
| `--fresh` | — | Ignore existing checkpoint and restart |

**`--in-place` mode — within the same project:**

```bash
cd my-angularjs-project
ng-migrate migrate-project --in-place
```

Structure created:
```
my-project/
├── src-angularjs-backup/   ← full backup of originals (created before any changes)
├── src-angular21/          ← files migrated to Angular 21
├── angular.json            ← generated/updated at root
├── tsconfig.json           ← generated/updated at root
└── package.json            ← updated with Angular 21 dependencies
```

**Default mode — separate folder (with `ng new`):**

Before migrating any file, the CLI automatically runs `ng new` to create the official Angular 21 structure. The AngularJS project files are then migrated into that scaffold.

```
../my-project-angular21/          ← created via `ng new`
├── src/
│   └── app/
│       ├── user.component.ts      ← migrated from original controller
│       ├── auth.service.ts        ← migrated from original service/factory
│       └── currency.pipe.ts       ← migrated from original filter
├── angular.json                   ← generated by Angular CLI
├── tsconfig.json                  ← generated by Angular CLI
├── package.json                   ← updated with project deps
└── MIGRATION_REPORT_<date>.md
```

**Migration phases (automatic ordering):**

| Phase | Type | Reason |
|---|---|---|
| 1 | Services & Factories | No dependencies in most cases |
| 2 | Filters → Pipes | Depend only on services |
| 3 | Directives & Components | Depend on services and pipes |
| 4 | Controllers | Depend on everything above |
| 5 | HTML Templates | Depend on components |
| 6 | Routing & Modules | Last — references all previous phases |

### `migrate-repo` — Migrate remote repository

Clones (in memory) a GitHub or GitLab repository and migrates all AngularJS files.

```bash
# Public GitHub
ng-migrate migrate-repo owner/repo

# Private GitHub
ng-migrate migrate-repo owner/repo --github-token ghp_...

# GitLab
ng-migrate migrate-repo group/project --gitlab-token glpat-...

# Self-hosted GitLab
ng-migrate migrate-repo group/project \
  --gitlab-token glpat-... \
  --gitlab-url https://gitlab.my-company.com

ng-migrate migrate-repo owner/repo --branch develop

# Custom output folder

# Migrate only part of the project
ng-migrate migrate-repo owner/repo --only "src/app/**"

# Dry-run: analyze without saving
ng-migrate migrate-repo owner/repo --dry-run

# Automatically create Pull Request after migration
ng-migrate migrate-repo owner/repo --create-pr
```

**Options:**

| `-b, --branch <branch>` | default branch | Branch to migrate |
| `-o, --output <folder>` | `./<repo>-angular21` | Output folder |
| `--concurrency <n>` | `3` | Files in parallel |
| `--dry-run` | — | Analyze without saving files |
| `--only <glob>` | — | Filter by glob |
| `--skip-deps` | — | Don't update `package.json` |
| `--create-pr` | — | Automatically create PR/MR in the repository |

### `watch` — Real-time file watcher

Monitors a folder and automatically migrates each saved AngularJS file, using AI and applying post-cleanup.

```bash
ng-migrate watch                      # Monitor current folder
ng-migrate watch src/app              # Monitor specific folder
ng-migrate watch src --out dist/ng21  # Set output folder
ng-migrate watch --only .js           # Monitor only .js files

```

**Options:**

| Option | Default | Description |
|-------|--------|-----------|
| `[folder]` | `.` (current) | Folder to monitor |

| `--dry-run` | — | Detect changes but don't run migration |

> `watch` uses `chokidar` internally. When a file is saved, it is immediately migrated with post-cleanup applied. Ideal for incremental refactoring.

## Advanced Features

### Vendor and minified file filtering

`ng-migrate` automatically ignores files that should not be migrated, **saving AI tokens** and speeding up the scan:

**Ignored patterns:**
- Minified files: `*.min.js`, `*.min.css`, `*.bundle.js`, `*.packed.js`
- Vendor folders: `vendor/`, `bower_components/`, `public/lib/`, `assets/lib/`, `assets/vendor/`, `static/lib/`
- Known libraries by name: jQuery, Bootstrap, Lodash, Underscore, Moment.js, `angular.js`, `angular-mocks.js`
- **Size limit:** files larger than **300 KB** are automatically skipped

These filters apply to both `scan` and `migrate-project`.

### Automatic post-cleanup

After each migration, `ng-migrate` applies **28 automatic correction rules** to the AI-generated code, ensuring Angular 21 compatibility:

| Category | Example corrections |
|-----------|-----------------------|
| **Obsolete APIs** | `angular.copy` → `structuredClone()`, `angular.extend` → `Object.assign()` |
| **Imports** | Removes duplicate or obsolete `@angular/core` imports |
| **Template syntax** | `ng-if` → `@if`, `ng-for` → `@for`, `ng-switch` → `@switch` |
| **Signals** | `this.value` → `this.value()` in signal contexts |
| **Decorators** | Ensures `standalone: true` on all components |

Post-cleanup runs automatically in `migrate-project`, `migrate`, and `watch`.

### Plugin system

`ng-migrate` supports a plugin system to customize migration behavior:

**Plugin structure** (`.js` or `.mjs` file):

```js
export default {
  // Additional skip rules (in addition to defaults)
  skipPatterns: [/my-legacy\.js$/],

  // Custom post-cleanup rules
  cleanupRules: [
    { pattern: /oldAPI\(/g, replacement: "newAPI(" }
  ],

  // Additions to the AI prompt
  promptAdditions: "Preserve JSDoc comments on all public methods.",

  // Hook executed after each file is migrated
  async postProcess(filePath, content) {
    return content.replace(/console\.log/g, "// console.log");
  }
};
```

**Usage:**

```bash
ng-migrate migrate-project --plugin ./my-plugin.mjs
```

## Recommended workflow

### Option A — Automated pipeline (`start`)

```bash
# 1. Configure AI (one time only)
ng-migrate config

# 2. Go to the project folder
cd my-angularjs-project

# 3. Start the full pipeline
ng-migrate start
# → Scans, shows the plan, asks for confirmation, and migrates automatically
```

### Option B — Manual control

```bash
# Step 1: Configure AI (one time only)
ng-migrate config

# Step 2: Go to the project folder
cd my-angularjs-project

# Step 3: Analyze the project
ng-migrate scan
# → Read the summary: complexity, phases, estimated hours, library report

# Step 4 (optional): Deep AI analysis
ng-migrate scan --ai --force

# Step 5: Check library migration report
ng-migrate libs

# Step 6: Preview the plan without executing
ng-migrate migrate-project --dry-run

# Step 7: Run the migration
ng-migrate migrate-project
# What happens automatically:
#   1. Checks/installs Angular CLI globally (if needed)
#   2. PHASE 1 — Scans the project: shows files, complexity, phase plan
#   3. PHASE 2 — Runs `ng new` to create the Angular 21 base project
#   4. AI migrates each file by phase (services → routing)
#   5. package.json is updated with correct deps
#   6. npm install is run in the final project

# In-place mode (within the same project, without ng new)
ng-migrate migrate-project --in-place

# Migrating directly from a remote repository
ng-migrate migrate-project --clone https://github.com/user/repo

# Resume interrupted migration (uses automatic checkpoint)
ng-migrate migrate-project
# To restart from scratch ignoring the checkpoint:
ng-migrate migrate-project --fresh

# Step 8: Review and fix errors
# The MIGRATION_REPORT_*.md report lists files with issues
cd ../my-project-angular21
ng serve
```

```bash
ng-migrate migrate-repo owner/repo --dry-run

# Migrate
```

## Files generated by scan

|---|---|

| `$http` | `HttpClient` with `inject()` |

| `ng-if` | `@if` (control flow) |
| `ng-for` / `ng-repeat` | `@for` (control flow) |
| `ng-switch` | `@switch` (control flow) |

## Access tokens

### GitHub
1. Go to: https://github.com/settings/tokens/new
2. Minimum scopes: `repo` (private) or `public_repo` (public)
3. For `--create-pr`: add full `repo` scope
4. Save with: `ng-migrate env set GITHUB_TOKEN ghp_...`

### GitLab
1. Go to: Profile → Access Tokens → Add new token
2. Minimum scopes: `read_repository`
3. For `--create-pr`: add `write_repository`, `api`
4. Save with: `ng-migrate env set GITLAB_TOKEN glpat-...`

### ARIA — Migration agent

The AI operates under the **ARIA** persona (*Angular Refactoring Intelligence Agent*), with absolute rules that ensure idiomatic, complete Angular 21 code:

| Rule | Description |
|---|---|
| **R1** | **Complete** code — no `// TODO`, placeholders or omitted sections |
| **R2** | `standalone: true` on 100% of Components, Pipes and Directives |
| **R3** | `signal()` for reactive state, `computed()` for derived values, `effect()` for side-effects |
| **R4** | `inject()` for dependency injection — constructor only when `super()` is required |
| **R5** | Modern template control flow: `@if`, `@for` (with `track`), `@switch`, `@defer` |
| **R6** | TypeScript strict — explicit types on public properties and parameters |
| **R7** | Consistent naming: `myCtrl` → `MyComponent`, `userSvc` → `UserService` |
| **R8** | No explanatory comments — self-documenting code |
| **R9** | Preserve 100% of original business logic |

### Model routing by complexity

The `MigrationOrchestrator` automatically selects the model tier based on the complexity and phase of each file:

| Tier | Used for |
|---|---|
| **fast** | Low-complexity files · Filters · Templates |
| **standard** | Medium complexity · Services · Directives · Controllers |
| **premium** | High complexity · Routing & Modules (phase 6) |

The cache (`.ng-migrate-cache.json`) uses SHA-256 content hash — unmodified files are not re-migrated.
